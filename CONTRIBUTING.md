# Contributing Guide
欢迎参与 snail-job-master 开源项目共建，无论提交 Bug 反馈、功能建议、文档优化或是代码 PR，都十分感谢你的付出。本文档统一规范社区贡献流程、代码标准与提交要求，请在参与开发前完整阅读。

## 一、前期准备
### 1. 基础环境
开发本项目需要提前准备对应环境：
- JDK 17+
- Maven 3.6 及以上
- MySQL 8.0 / PostgreSQL 任意主流版本
- Git 工具

### 2. 仓库 Fork 与本地克隆
1. 访问项目 GitHub 主页，点击右上角 Fork，将仓库复制至个人账号；
2. 克隆个人 Fork 仓库到本地：
```bash
git clone https://github.com/你的用户名/snail-job-master.git
cd snail-job-master
```
3. 绑定上游原仓库，用于同步最新代码：
```bash
git remote add upstream https://github.com/SCIENCE-FRESHMEN/snail-job-master.git
```

## 二、Issue 规范（所有变更前置要求）
任何代码修改、新增功能、Bug 修复，**必须先创建 Issue**，禁止无 Issue 直接提交 PR。
1. Issue 分类标签
    - `bug`：程序运行异常、逻辑错误、报错崩溃等问题
    - `feature`：全新功能需求、能力扩展
    - `enhancement`：现有功能优化、性能提升、代码重构
    - `docs`：文档修正、补充、翻译优化
    - `question`：使用疑问、部署咨询
2. Issue 书写要求
    - Bug 类：清晰复现步骤、运行环境、异常堆栈、预期结果与实际结果；
    - 功能/优化类：说明使用场景、现有痛点、实现思路、预期效果；
3. 多人协作：可在 Issue 下方留言认领，避免多人重复开发同一需求。

## 三、分支管理规范
项目长期维护两条核心分支：
1. `main`：稳定发布分支，仅合并经过测试、审核完成的代码，禁止直接提交；
2. `dev`：开发主分支，所有功能迭代、Bug 修复均基于该分支开发。

### 分支创建规则
从本地 `dev` 分支拉取新功能分支，分支命名统一格式：
- Bug 修复：`fix/issue-编号-简短描述`
- 新增功能：`feature/issue-编号-简短描述`
- 代码重构/优化：`optimize/issue-编号-简短描述`
- 文档修改：`docs/issue-编号-简短描述`

示例：
```bash
# 同步上游最新代码
git fetch upstream
git checkout dev
git merge upstream/dev

# 创建修复分支
git checkout -b fix/issue-102-retry-timeout
```

## 四、代码开发规范
### 1. Java 编码标准
1. 统一遵循阿里巴巴 Java 开发手册规范；
2. 类、接口、核心方法必须添加标准 JavaDoc 注释，说明功能、入参、返回值、异常；
3. 复杂业务逻辑、算法代码增加行内注释，便于其他开发者阅读；
4. 禁止硬编码配置、魔法数字，统一放入常量类管理；
5. 新增功能同步补充单元测试，修复 Bug 需增加对应复现测试用例；
6. 所有模块编译无报错、单元测试全部通过。

### 2. 配置与文档
1. 新增配置项同步更新 doc 目录下使用文档；
2. 数据库表结构变更，补充 SQL 脚本与更新说明；
3. 控制台页面、注解使用示例同步补充代码样例。

### 3. 禁止提交内容
提交代码前过滤以下文件，严禁推入仓库：
- IDE 配置文件（.idea、.vscode、*.iml）
- 本地环境配置、数据库连接明文账号密码
- 编译产物 target、日志文件、缓存文件
- 临时测试代码、注释掉的废弃代码

## 五、Git 提交规范
每次 commit 信息统一格式，便于日志检索与版本追溯：
```
[类型] #Issue编号 简短中文描述
详细补充说明（可选，多行）
```
类型取值：`fix` / `feature` / `optimize` / `docs` / `refactor`

示例：
```
[fix] #102 修复重试任务超时未触发死信问题
优化任务执行超时判断逻辑，增加超时标记，超时任务自动转入死信队列
```

### 提交操作建议
1. 单次功能开发尽量压缩少量 commit，避免几十条零散提交；
2. 推送前拉取上游 dev 分支，解决代码冲突，保证无合并冲突；
```bash
git fetch upstream
git merge upstream/dev
# 解决冲突后重新提交
```

## 六、Pull Request（PR）提交流程
1. 将本地分支推送至个人 Fork 仓库：
```bash
git push origin fix/issue-102-retry-timeout
```
2. 进入 GitHub 个人仓库页面，发起 Pull Request，目标分支选择上游仓库 `dev`；
3. PR 标题与描述规范：
    - 标题：对应 Issue 核心内容；
    - 描述：关联 Issue 编号、修改范围、实现思路、自测验证结果；
    - 截图/日志：界面改动、Bug 修复可附上运行截图辅助审核。
4. PR 审核要求
    - CI 流水线全部通过（编译、单元测试、代码格式检查）；
    - 维护者进行代码评审，根据评论修改代码并推送更新；
    - 重大功能改动需多名维护者共同审核通过后方可合并。

## 七、社区贡献身份说明
1. 普通贡献者：提交有效 PR 并成功合并，你的名称将记录在项目贡献者列表；
2. Committer 准入条件：
    - 累计完成至少 3 个高质量合并 PR；
    - 主动参与 Issue 答疑、协助新人解决开发问题；
    - 熟悉项目核心调度、重试底层逻辑；
    满足条件后可申请 Committer 权限，参与分支合并、Issue 管理工作。
3. 长期维护要求：Committer 需保持活跃，定期处理社区反馈与代码审核。

## 八、开源交流与问题反馈
1. 代码问题、功能需求统一在 GitHub Issues 提交；
2. 使用部署类疑问可在 Issue 下留言讨论；
3. 请勿通过私信单独询问基础使用问题，公开讨论可帮助更多使用者。

## 九、开源协议声明
提交至本项目的所有代码、文档资源，均默认遵循项目开源协议，提交者确认拥有代码完整版权，无第三方版权纠纷，允许项目永久开源使用与迭代分发。
